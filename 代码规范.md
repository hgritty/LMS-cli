## 1. 代码格式规范

### 1.1 基本规则

| 规则 | 说明 |
|------|------|
| 缩进 | 使用 **4 个空格**（不使用 Tab） |
| 行宽 | 每行最多 **80-100 个字符** |
| 空格 | `if`、`for`、`while`、`switch` 后面加一个空格 |
| 运算符 | 运算符两侧加空格（`=`, `+`, `-`, `*`, `/`, `==` 等） |
| 大括号 | 左大括号不换行，右大括号独占一行 |
| 空行 | 函数之间空一行，逻辑块之间空一行 |

### 1.2 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件名 | 小写 + 下划线（snake_case） | `library_management.cpp` |
| 结构体/类 | 大驼峰（PascalCase） | `Book`, `LibraryManager` |
| 函数名 | 小驼峰（camelCase） | `addBook()`, `listBooks()` |
| 变量名 | 小驼峰（camelCase） | `bookTitle`, `nextId` |
| 常量 | 全大写 + 下划线（UPPER_SNAKE_CASE） | `MAX_BOOKS` |
| 全局变量 | 描述性名称（可加 `g_` 前缀） | `g_books` |
| 布尔变量 | 用 `is` / `has` / `can` 开头 | `isBorrowed`, `hasCopies` |

### 1.3 文件头注释

每个 `.cpp` 和 `.h` 文件都应有文件头注释：

```cpp
// =========================================
// 文件: library.cpp
// 描述: 图书管理系统 - 核心功能实现
// 作者: hgritty
// 日期: 2026-06-29
// 版本: v1.0
// =========================================
```

### 1.4 函数注释

使用 `/** */` 风格，说明功能、参数和返回值：

```cpp
/**
 * 添加一本新书到系统
 * 
 * 用户输入书名和作者，系统自动分配唯一编号
 * 新书默认状态为"在库"（未被借出）
 * 
 * @param 无
 * @return 无
 */
void addBook() {
    // ...
}
```

### 1.5 行内注释

- 复杂逻辑必须加注释
- 注释写在代码**上方**或**同行右侧**
- 同行注释与代码之间至少空 2 个空格

```cpp
int nextId = 1;       // 下一本书的编号（从 1 开始）

// 遍历所有书，查找匹配的 ID
for (const Book& b : books) {
    if (b.id == id) {
        // 找到目标书籍
    }
}
```

### 1.6 代码示例对比

#### ✅ 好的写法

```cpp
int calculateAverage(const vector<int>& numbers) {
    if (numbers.empty()) {
        return 0;
    }

    int sum = 0;
    for (int num : numbers) {
        sum += num;
    }

    return sum / numbers.size();
}
```

```cpp

if(numbers.empty())return 0;
int sum=0;
for(int num:numbers){sum+=num;}
return sum/numbers.size();}
```

## 2. Git 提交规范

### 2.1 提交信息格式

```
<类型>(<范围>): <简短描述>

<详细描述（可选）>

<脚注（可选）>
```

### 2.2 提交类型

| 类型 | 说明 | 使用场景 |
|------|------|---------|
| `feat` | 新增功能 | 添加新功能、新模块 |
| `fix` | 修复 Bug | 修复逻辑错误、崩溃、异常 |
| `docs` | 文档/注释 | 更新 README、注释、文档 |
| `style` | 代码格式 | 调整缩进、空格、删除多余空行（不影响逻辑） |
| `refactor` | 重构 | 重命名、提取函数、优化结构（不改变功能） |
| `test` | 添加测试 | 添加或修改测试用例 |
| `chore` | 构建/工具 | 修改 .gitignore、编译配置、依赖 |
| `init` | 初始提交 | 第一次提交项目 |

### 2.3 范围（可选）

范围表示修改的模块或组件，放在括号内：

```
feat(book): 添加借书功能
fix(ui): 修复列表表格对齐
docs(readme): 更新安装说明
```

### 2.4 简短描述规则

| 规则 | 说明 |
|------|------|
| 语言 | 中文或英文（保持一致） |
| 时态 | 使用**动词原形**（"添加" 不是 "添加了"） |
| 首字母 | **小写** |
| 结尾 | **不加句号** |
| 长度 | 最多 **50 个字符** |

### 2.5 详细描述规则（可选）

- 说明**改了什么**和**为什么改**
- 多条修改用 **项目符号**（`-`）列出
- 每行最多 **72 个字符**

### 2.6 提交示例

#### 简单提交

```bash
git commit -m "feat: 添加借书功能"
```

#### 带详细描述的提交

```bash
git commit -m "feat: 添加数据持久化支持

- 退出时自动保存数据到 books.txt
- 启动时自动从 books.txt 加载数据
- 使用 | 作为字段分隔符"
```

#### 带范围的提交

```bash
git commit -m "feat(book): 添加借书功能"
```

#### 完整示例

```bash
git commit -m "fix(ui): 修复列表表格列宽问题

- 使用 setw() 替代 printf 实现对齐
- 中文书名不再导致表格错位
- 统一表格边框样式"
```

### 2.7 不好的示例

```bash
# ❌ 太模糊，不知道改了啥
git commit -m "更新代码"

# ❌ 同上
git commit -m "修改"

# ❌ 没有说明改动内容
git commit -m "123"

# ❌ 过于啰嗦，且没有类型
git commit -m "我今天改了借书功能里面的一些bug还有格式问题"
```

### 2.8 你项目的提交记录示例

| 你做的事 | 建议的 commit |
|---------|---------------|
| 创建项目 | `init: 创建图书管理系统项目` |
| 添加 `.gitignore` | `chore: 添加 .gitignore` |
| 实现添加功能 | `feat: 实现添加图书功能` |
| 实现列表功能 | `feat: 实现查看所有图书功能` |

| 实现还书功能 | `feat: 实现还书功能` |

| 添加函数注释 | `docs: 为所有函数添加注释` |
| 添加代码规范文档 | `docs: 添加代码规范和 Git 提交规范` |

## 4. 代码审查清单

提交代码前，检查以下内容：

### 4.1 格式检查

- [ ] 缩进使用 4 个空格
- [ ] 行尾没有多余空格
- [ ] 文件末尾有空行
- [ ] 运算符两侧有空格
- [ ] 大括号位置正确

### 4.2 命名检查

- [ ] 函数名用动词开头
- [ ] 变量名有意义（不叫 `a`、`b`、`tmp`）
- [ ] 结构体用大驼峰
- [ ] 常量用全大写

### 4.3 注释检查

- [ ] 每个函数有功能说明
- [ ] 复杂逻辑有行内注释
- [ ] 注释与代码一致（不同步的注释比没有注释更糟糕）

### 4.4 Git 检查

- [ ] commit 类型正确
- [ ] commit 描述清晰
- [ ] 本次提交只做一件事

## 5. 总结

| 类别 | 核心原则 |
|------|---------|
| **格式** | 统一、清晰、可读 |
| **命名** | 见名知意，保持一致 |
| **注释** | 解释"为什么"，不解释"是什么" |
| **Git** | 每次提交只做一件事，描述清晰 |
| **设计** | 高内聚、低耦合、易扩展 |
